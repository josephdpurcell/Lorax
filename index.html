    Environmental

    • Point: The monetary analogy communicates the wrong idea about how technical debt is measured and how it impacts business.
    • Why is it a problem to compare tech to money?
    • Money is a common exchange of value; translation code is a common exchange of effort
    • Logical conclusions for a business person:
      • tech debt = LOC per person per hour * # of persons
      • If I know how much effort per line of code I know the cost.
      • I should be able to ask questions like:
        1. Where is the technical debt?
        2. How much do we owe?
        3. How much is it costing us in interest repayments?
        4. How long will it take to repay?
    • These are easy questions in the financial world because we have bank statements and balance sheets readily available to quantify monetary debt, and there is an exact tradeoff between spending income on debt versus new assets.
    • In the tech world, we are still defining what good code is and there are contextual nuances that prevent consensus.
    • When pitting working on tech debt versus new features, clients expect to know the cost-benefit analysis, because it must compare to money.
    • Monetary debt is discrete and quantifiable, whereas technical debt is connected and qualitative.
    • It's ok to default on tech debt, but not money. (source)
    • Monetary debt has a schedule for repayment, but not tech debt. (source)
